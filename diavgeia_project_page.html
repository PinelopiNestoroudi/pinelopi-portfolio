Project Overview

The tool was built to turn raw API documentation and repeated manual querying into a more structured exploration workflow. Instead of making one-off calls and manually tracking parameters, the interface groups the main use cases into dedicated views and adds logic to make public-sector data retrieval more practical.

What the tool does

Three focused exploration workflows

  • Loads and displays organizations, including a narrower university-focused view.
  • Queries decisions for a selected organization and date window, with support for repeated exploration across periods.
  • Retrieves active organizational units for a selected organization and presents them in a usable table view.
  • Uses browser-side logic to loop through paginated responses and collect complete result sets where needed.
  • Adds UI-side filtering for cases where the API does not provide the desired filtering directly.
  • API querying
  • Pagination handling
  • UI filtering
  • Public data exploration
Why it matters

A usability layer over raw API access

The value of the project is not just the API call itself, but the surrounding logic: helping the user choose valid date ranges, move to the next period quickly, inspect revoked decisions, identify private-data cases, and retrieve organizational unit information without manually reconstructing calls every time.

Practical focus

This project shows applied API documentation reading, query construction, UI logic, and browser-side orchestration for a real public-data use case.

Key Capabilities

The project groups related tasks into guided tabs and adds lightweight validation and automation so users can focus on exploration rather than reconstructing the request logic manually.

Organizations

General and university views

The organizations section retrieves all available organizations and also provides a narrower university-focused view, making the dataset easier to explore for the intended use case.

Decisions

Date-aware query workflow

The decisions section lets the user query by organization and date range, count all decisions, isolate revoked decisions, and detect cases involving private data through UI-side filtering.

Units

Organizational structure inspection

The organizational-units section retrieves active units for a selected organization and presents the result in a clean, usable table with identifiers and parent relationships.

Implementation Notes and Outcome

The project is intentionally lightweight and browser-based, which made it practical to build and demonstrate, while also revealing where a fuller backend implementation would add performance, control, and extensibility.

What this project shows

Applied API logic with user-facing structure

  • Reading API documentation and turning constraints into usable interface behavior.
  • Handling paginated results instead of limiting the output to one response page.
  • Adding UI logic where the API does not directly support the desired filter.
  • Designing a lightweight browser interface around repeated public-data exploration tasks.
Implementation notes

Useful prototype with clear backend potential

Current approach

The implementation is client-side and relies on browser-side processing for calls, aggregation, and filtering. It demonstrates the logic clearly, but is still prototype-oriented.

Future evolution

A backend version could improve performance, reduce browser constraints, and support a more robust and scalable exploration experience.

The project brings together public-data exploration, API constraints, and usability-focused interface logic in one lightweight workflow.
